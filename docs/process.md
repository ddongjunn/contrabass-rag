# process.md — 단계별 개발·검수 프로세스 (불변 규약)

> 요구사항 [`requirements.md`](requirements.md) · 설계 [`architecture.md`](architecture.md) · 계획 [`plan.md`](plan.md).
> **이 문서는 "어떻게 일을 진행하는가"의 규약이다.** `plan.md`의 Phase를 실행할 때 매번 이 절차를 따른다.
> 합의된 규정(2026-06-04): ① Phase별 **개발→검수 사이클**, ② 너무 큰 Phase는 **서브 Phase로 분할**해 안전하게,
> ③ 검수는 **Claude 체크 + 사용자 체크 둘 다 통과**해야 다음으로.

---

## 0. 황금 규칙 (매 작업 앞에서 재확인)

1. **Phase 순서 고정** — `plan.md` Phase 0→7 순차. 이전 Phase **DoD + 검수 통과** 전에 다음 Phase 코드 금지.
2. **2차 기능 끌어오지 않기** — `plan.md §2차(착수 금지)` 항목은 1차에 절대 손대지 않는다.
3. **불변식 우선** — `CLAUDE.md §절대 불변식` 8개를 어기는 코드는 검수 자동 실패.
4. **설정 외부화** — 임계값·모델·top-k 등은 `application.yml`(`app.*`) + `AppProperties`만. 하드코딩 = 실패.
5. **Surgical** — 요청·해당 Phase 범위 밖 코드는 건드리지 않는다.

---

## 1. Phase 실행 사이클 (매 Phase 동일)

```
[1] 착수 전 정렬   → [2] 서브 분할 판단 → [3] 개발 → [4] Claude 검수 → [5] 사용자 검수 → [6] 커밋·다음 Phase
        │                                                   │              │
        └─ 불변식·DoD·범위 확인                              └─실패 시 [3]으로 되돌아감(다음 Phase 진입 금지)
```

### [1] 착수 전 정렬 (Claude가 먼저 제시 → 사용자 승인)
- 해당 Phase의 **DoD**(plan.md)와 **건드릴 모듈/파일 목록**을 한눈에 제시.
- 관련 **불변식**과 **외부 의존**(DB·OpenAI·Slack 토큰 필요 여부)을 명시.
- 가정·불확실점이 있으면 **코딩 전에 질문**(CLAUDE.md §1).

### [2] 서브 Phase 분할 판단 (안전장치)
한 Phase를 한 번에 짜기엔 크거나 불안정하면 **반드시 서브 Phase로 쪼갠다.** 분할 신호:
- 새 파일/클래스 **5개 이상** 또는 외부 I/O(DB·OpenAI·Slack)가 **2종 이상** 동시 도입.
- 인터페이스 + 실제 외부호출 구현 + 정책이 **한 덩어리**로 엮일 때 → 인터페이스/스텁 먼저, 실구현 나중.
- 검수 한 번에 확인할 항목이 **너무 많아** 회귀 원인 추적이 어려워질 때.

**분할 원칙:** 각 서브 Phase도 **자체적으로 빌드되고 검수 가능한 최소 단위**여야 한다(절반 짜다 만 상태 금지).
분할하면 `plan.md`에 `Phase N-a / N-b …`로 체크박스를 추가하고 그 단위로 사이클을 돈다.

### [3] 개발
- 최소 코드로 DoD만 충족(CLAUDE.md §2). 새 기능·추상화·설정 금지.
- 검증 가능한 성공조건 우선: "validation 추가" → "잘못된 입력 테스트 작성 후 통과"(CLAUDE.md §4).

### [4] Claude 검수 — **내가 코드/빌드로 확인하는 것** (자동·정적)
다음을 모두 통과해야 사용자 검수로 넘긴다. 하나라도 실패면 [3]으로.

- [ ] **빌드/컴파일 성공** (`./gradlew build` 또는 `compileKotlin`).
- [ ] **테스트 통과** (해당 Phase에 추가한 테스트 + 기존 테스트 그린).
- [ ] **불변식 정적 점검**: 임베딩 모델/차원(text-embedding-3-small/1536) 고정 · 질문 임베딩 1회 재사용 ·
      파이프라인 순서(임베딩→검색→생성; **캐시 단계는 고도화 보류**) · 출처(title/page) 표기 · `documents` 읽기 전용.
- [ ] **설정 외부화**: 새 임계값/모델/경로가 코드에 박히지 않고 `application.yml`+`AppProperties` 경유.
- [ ] **범위 고정**: 2차 기능 미혼입, 다음 Phase 코드 미선반영.
- [ ] **패키지 구조** 준수(`architecture.md §2` 모듈×4계층, 단일 진입점=`ChatService`).
- [ ] **Surgical**: diff의 모든 줄이 이번 Phase 요청에 직접 대응(무관한 리팩터/포맷 변경 없음).
- [ ] **시크릿**: 키/토큰이 코드·yml에 평문으로 들어가지 않음(환경변수만).

→ 통과 시 Claude는 **"무엇을·어떻게 검증했는지 + 사용자 검수 항목"**을 요약해 보고한다.

### [5] 사용자 검수 — **네(사용자)가 직접 확인해야 하는 것** (런타임·외부·비용·보안)
Claude가 대신 할 수 없는, **실제 환경/판단**이 필요한 항목. 사용자가 OK해야 다음 Phase로 간다.

- [ ] **실제 실행/DoD 시연**: 해당 Phase DoD를 실 환경에서 확인
      (예: `GET /actuator/health`=UP, `POST /api/chat` 응답, Slack 멘션→스레드 답변).
- [ ] **외부 의존 연결**: 사내망 **DB(pgvector) 연결**, **OpenAI API 키**, **Slack 토큰** 주입·동작.
- [ ] **비용/호출 검증**: 케이스별 **LLM 호출 횟수**(검색실패=0회, 차단=0회) 실측. (캐시 히트=0회는 고도화)
- [ ] **보안 승인**: 사내 문서·질의의 **OpenAI 외부 전송** 정책 승인(requirements.md §5) 확인.
- [ ] **데이터 계약**: 외부 색인팀과 합의된 값(테이블명·metadata 키 `title`/`page`·차원) 실제 일치.
- [ ] **DoD 최종 승인**: "이 Phase 완료" 판단은 사용자가 내린다.

### [6] 커밋 · 다음 Phase
- 두 검수 통과 후에만 커밋(요청 시). `plan.md` 체크박스 갱신. 다음 Phase [1]로.
